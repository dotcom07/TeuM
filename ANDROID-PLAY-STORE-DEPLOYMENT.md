## 1. 배포 목표

직장인이 업무 시간 중 물 한 모금과 가벼운 스트레칭을 놓치지 않도록 돕는 **직장인 건강 반복 알람**으로 Google Play에 출시한다.

- 서비스명: `틈새움 TeuM`
- 앱 아이콘 표기: `TeuM`
- 핵심 한 줄: `일하는 나를 위한 1분`
- 영문 보조 의미: `One minute for a healthier workday.`
- 수익화: 없음 (초기)
- 계정·서버·광고·분석 SDK: 없음 (초기)
- 외부 데이터 수집·전송: 없음. 설정과 사용자가 선택적으로 남긴 O/X 기록만 기기 내 저장.

### 포지셔닝 원칙

제품의 건강 목적과 알람이라는 핵심 기능을 같은 문장 안에서 정직하게 전달한다.

> 사용자가 정한 업무 시간에 반복 알람을 울려, 물 한 모금과 가벼운 스트레칭을 챙기게 돕는 직장인 건강 앱.

| 대상 | 먼저 전달할 내용 | 피할 표현 |
| --- | --- | --- |
| 일반 사용자 | 스트레칭·물 마시기를 챙기는 건강 습관 | 단순 알람시계, 생산성 감시, 운동 강요 |
| Play 권한 심사 | 사용자가 직접 일정·요일·간격을 설정하는 반복 알람이 핵심 기능 | 백그라운드에서 임의로 보내는 건강 푸시 |
| 건강 앱 심사 | 활동을 권하고 선택형 실천 기록을 제공하지만 건강 상태는 측정하지 않음 | 치료·예방·교정·의학적 효과 보장 |

스토어에서는 `건강 알람`을 대표 카테고리 문구로 사용한다. `알람`만 단독으로 강조하지 않고 항상 `직장인`, `스트레칭`, `물 한 모금`, `건강` 중 하나 이상과 함께 사용한다.

## 2. 전체 화면 반복 알람 정책

현재 Android 앱은 `SCHEDULE_EXACT_ALARM`과 `USE_FULL_SCREEN_INTENT`를 사용한다. 전자는 사용자가 허용하는 정확한 알람 접근이고, 후자는 Android 14 이상에서 별도 특별 접근 권한이다.

### 정책 리스크

`USE_FULL_SCREEN_INTENT`의 기본 허용 대상은 핵심 기능이 **알람 또는 통화**인 앱이다. 틈새움은 사용자가 정한 업무 시간과 요일에 사용자 지정 간격(기본 60분)의 알람을 예약하고, 알람이 울리면 1분 화면을 여는 **직장인 건강 반복 알람 앱**으로 정의한다. 반복 알람 전달이 빠지면 서비스의 핵심 기능이 성립하지 않는다.

Play Console 선언은 이 실제 동작을 기준으로 제출한다. 다만 최종 적격성은 Google 심사 결과에 따르므로 자동 승인을 전제하지 않는다. Android 14 이상에서 권한이 자동 부여되지 않거나 사용자가 끈 경우에는 특별 접근 설정으로 안내하고, 거부 상태에서도 높은 우선순위의 일반 알림으로 핵심 행동을 수행할 수 있어야 한다.

또한 Android 12 이상은 브로드캐스트 리시버에서 Activity를 직접 여는 notification trampoline을 제한한다. 따라서 스토어용 빌드는 리시버가 Activity를 직접 시작하는 방식에 의존하지 않고, 시스템 `fullScreenIntent`와 알림 탭을 이용해야 한다.

### 제품 결정

**전체 화면 반복 알람 유지형으로 확정한다.**

- `USE_FULL_SCREEN_INTENT`와 `SCHEDULE_EXACT_ALARM`을 유지한다.
- 온보딩과 설정에서 전체 화면 알람 특별 접근의 목적을 설명한다.
- Android 14 이상에서는 `canUseFullScreenIntent()`로 허용 여부를 확인한다.
- 허용되지 않았으면 `ACTION_MANAGE_APP_USE_FULL_SCREEN_INTENT` 설정으로 이동할 수 있게 한다.
- 잠금 상태에서는 시스템 `fullScreenIntent`, 앱 실행 중에는 앱 내부 타이머로 1분 화면을 연다.
- 권한 거부 시 일반 알림을 유지하고, 알림을 누르면 1분 화면을 연다.
- 브로드캐스트 리시버에서 Activity를 직접 실행하지 않는다.
- 화면이 켜진 상태의 표시 방식은 OS가 결정할 수 있으므로 모든 기기에서 강제 전체 화면을 보장한다고 홍보하지 않는다.

### Play Console 전체 화면 인텐트 선언 초안

| 항목 | 제출 내용 |
| --- | --- |
| 핵심 기능 | 사용자가 설정한 업무 시간과 요일에 사용자 지정 간격(기본 60분)으로 실행되는 반복 알람 |
| 권한 필요 이유 | 화면이 잠겨 있거나 꺼진 상태에서도 예약한 1분 건강 알람을 즉시 인지하게 하기 위해 사용 |
| 사용자 제어 | 업무 시간·요일·알림 켜기·일시정지·무음/진동을 사용자가 직접 설정 |
| 권한 거부 대체 | 높은 우선순위 일반 알림을 남기고, 탭하면 1분 화면을 실행 |
| 심사용 증빙 | 알람 예약 → 화면 잠금 → 예정 시각 전체 화면 실행 → 5분 뒤/O/X 응답 → 다음 알람 예약 흐름을 영상으로 제출 |

`SCHEDULE_EXACT_ALARM`은 사용자가 허용하는 방식이므로 현재 목적에 더 적합하다. “정확한 60분 리마인더”가 사용자에게 보이는 핵심 기능일 때만 유지하고, 권한이 꺼진 경우에는 다음 알림이 늦을 수 있음을 앱 안에서 안내한다.

## 4. Play Console 앱 콘텐츠 작성안

### 4.1 앱 분류와 대상

| 필드 | 제안값 | 근거 |
| --- | --- | --- |
| 앱 | 앱 | 게임 아님 |
| 카테고리 | 건강 및 피트니스 | 물·스트레칭 습관을 위한 직장인 건강 알람 |
| 대상 연령 | 18세 이상 중심 | 직장인 대상, 아동용 기능 없음 |
| 광고 포함 | 아니요 | 현재 광고 SDK·광고 노출 없음 |
| 앱 액세스 | 제한 없음 | 로그인·유료벽·지역 제한 없음 |
| 콘텐츠 등급 | IARC 설문 완료 | 폭력·성·도박·광고 없는 일반 웰니스 앱 |
| 건강 앱 선언 | `Health and fitness > Activity and Fitness` | 가벼운 스트레칭 활동을 제안하고 선택형 실천 기록을 제공함. 의료 기능은 선택하지 않음 |

건강 앱 선언의 실제 선택지는 Play Console의 최신 설문 문구를 우선한다. 틈새움은 건강 상태나 물의 양을 측정·추정하지 않는다. 선택형 O/X 기록은 사용자가 직접 남긴 리마인더 응답이며, 진단·치료·질병 예방을 주장하지 않는다. 스토어 문구도 “치료”, “예방”, “의학적으로 보장” 같은 표현을 사용하지 않는다.

### 4.2 Data safety 작성안

선택형 기록까지 구현한 출시 버전의 목표 답변 초안이다. 제출 직전에 실제 release AAB와 포함 SDK를 기준으로 다시 검토한다.

| 질문 | 답변 초안 |
| --- | --- |
| 앱이 필수 사용자 데이터를 수집/공유하는가 | 아니요 |
| 기기 밖으로 전송되는 사용자 데이터가 있는가 | 아니요 |
| 데이터가 기기 내에서만 처리되는가 | 예. 업무 시간·요일·알림 방식과 선택형 O/X 기록만 기기에 저장 |
| 개인정보처리방침 URL | [Notion 공개본](https://aluminum-language-c6a.notion.site/TeuM-3a447d4c6bf180f7a6f6e221089a5012?pvs=73) 연결 완료 |
| 계정 삭제 | 해당 없음. 계정 기능 없음 |

중요: 데이터 수집이 없더라도 Data safety 양식과 공개 개인정보처리방침 URL은 필요하다. 향후 분석, 크래시 리포팅, 푸시, 광고, Health Connect, 로그인 SDK를 추가하면 해당 SDK의 전송 데이터도 선언해야 한다.

### 4.3 개인정보처리방침 필수 내용

공개 HTTPS 페이지와 앱 내 `개인정보` 화면에서 동일한 요지를 제공한다.

1. 운영자/개발자명, 문의 이메일
2. 앱이 기기 내에 저장하는 정보: 업무 시간, 요일, 알림/진동 설정, 일시정지 상태, 기록 모드를 켠 경우 O/X 응답과 응답 시각
3. 서버 전송·공유·판매가 없다는 사실
4. 저장 위치와 삭제 방법: 앱의 `기록 모두 삭제`, 시스템의 앱 데이터 삭제 또는 앱 삭제
5. 알림·정확한 알람·진동 권한의 목적
6. 변경일과 정책 변경 안내 방식

```text
• 5분 뒤에 다시, 챙겼어요, 이번엔 넘길게요
• 원할 때만 켜는 기기 내 기록 달력

계정이 필요하지 않습니다.
설정과 선택형 O/X 기록은 기기에만 저장되며 서버로 전송되지 않습니다.

틈새움은 의료 서비스나 진단 도구가 아니며 건강 상태를 측정하지 않습니다.
```

### 그래픽 자산

| 자산 | Play 요구사항 | 현재 상태 | 조치 |
| --- | --- | --- | --- |
| 스토어 앱 아이콘 | 512×512, 32-bit PNG, 1 MB 이하, alpha 가능 | `store-assets/android/teum-play-icon-512.png` 출력 완료 (272 KB) | 업로드 전 최종 시각 검수 |
| Feature graphic | 정확히 1024×500, JPEG 또는 24-bit PNG, alpha 없음 | `store-assets/android/teum-feature-graphic-1024x500.png` 출력 완료 (662 KB) | 업로드 전 최종 시각 검수 |
| 휴대전화 스크린샷 | 최소 2장 권장, 실제 앱 사용 모습 | 미제작 | 4~6장 제작 |
| 7인치/10인치 스크린샷 | 태블릿 노출을 원할 때 | 미제작 | v1은 제외 가능 |
| 프로모션 영상 | 선택 | 미제작 | v1은 제외 |

Feature graphic은 로고 전체를 크게 반복하지 않는다. 앱 아이콘과 함께 노출될 수 있으므로, 로고의 핵심 심볼을 작게 쓰고 중앙에 “직장인의 책상 → 1분의 틈 → 편안한 복귀” 장면을 둔다. 아주 작은 텍스트·세부 UI·과도한 게임 연출은 피한다.

### 스크린샷 시나리오 (5장)

1. **다음 틈까지** — 카운트다운과 진행 미터, 오늘의 업무 리듬
2. **1분의 틈** — 큰 60초 타이머와 물·가벼운 움직임 안내
3. **내 방식으로 알림** — 무음, 가볍게, 또렷하게, 확실하게
4. **업무에 맞춘 리듬** — 업무 시간·요일·잠시 멈춤 설정
5. **원할 때만 기록** — 날짜별 챙긴 횟수와 시간대별 O/X 기록

스크린샷은 실제 Android 앱 화면만 사용한다. 아이폰 프레임, 허위 알림, 의료 효과 수치, 랭킹·별점·다운로드 유도 문구는 넣지 않는다.

## 6. 기술 릴리스 계획

### P0 — 스토어 업로드 전 필수

- [x] 전체 화면 반복 알람 유지형으로 제품 결정을 확정한다.
- [x] 브로드캐스트 리시버가 Activity를 직접 여는 동작을 제거하고 시스템 `fullScreenIntent`를 사용한다.
- [x] 불필요 권한 3개(`READ_EXTERNAL_STORAGE`, `WRITE_EXTERNAL_STORAGE`, `SYSTEM_ALERT_WINDOW`)를 제거한다.
- [x] release signing을 debug keystore에서 분리하고 `keystore.properties.example` 템플릿을 만든다.
- [ ] Play App Signing을 활성화하고 업로드 키를 안전한 비밀 저장소에 보관한다.
- [ ] `versionCode=1`, `versionName=0.1.0`의 최초 release AAB를 생성한다.
- [x] 개인정보처리방침을 공개 HTTPS URL로 연결하고, 앱 설정 화면에 열기 버튼을 넣는다.
- [x] 스토어용 512×512 아이콘과 1024×500 feature graphic을 출력한다. 스크린샷 5장은 별도 제작한다.
- [ ] 실제 기기에서 무음·진동 1/3/5회 펄스(기기 무음 모드 포함), 알림 권한 거부, 정확한 알람 권한 거부, 잠금 화면, 앱 종료 상태를 각각 검증한다.
- [ ] 접근성 점검: TalkBack 레이블, 48dp 이상 터치 타깃, 색만으로 상태를 전달하지 않는지 확인한다.
- [ ] 선택형 기록 구현 후 `가볍게 사용`에서 응답 미저장, `기록 남기기`에서 O/X·응답 없음 저장, 날짜 집계, 전체 삭제를 검증한다.
- [ ] 선택형 기록을 출시 버전에 포함하기 전에 공개 개인정보처리방침에 기기 내 O/X 기록과 삭제 방법을 추가한다.

### P1 — 내부 테스트

1. `bundleRelease`로 AAB를 생성한다.
2. Play Console의 **Internal testing** 트랙에 업로드한다.
3. 최소 3개 기기군에서 테스트한다: Samsung One UI, Pixel Android 14+, 저가형 Android 12/13.
4. 테스트 항목: 권한 허용/거부, 업무 시간 안팎, 5분 뒤, O/X/응답 없음, 기록 모드 켜기·끄기·삭제, 화면 켜짐/잠금, 앱 강제 종료 후 알림, 재부팅 후 예약 복구.
5. Play pre-launch report의 크래시, 권한, 호환성 경고를 수정한다.

### P2 — 비공개 테스트 및 생산 권한

신규 개인 개발자 계정(2023-11-13 이후 생성)이면, production 접근 전 **비공개 테스트에서 최소 12명이 14일 연속 opt-in** 상태여야 한다. 조직 계정 또는 기존 개인 계정의 적용 여부는 Play Console Dashboard에서 확인한다.

테스터에게 아래 피드백을 받는다.

- 알림이 업무 흐름을 과도하게 끊는가?
- 잠금 화면·회의 중·다른 앱 사용 중 어떻게 보이는가?
- 진동 1·3·5회 펄스가 서로 구분되는가? 기기 무음 모드에서도 전달되는가?
- `5분 뒤에 다시`, `네, 챙겼어요`, `이번엔 넘길게요`의 의미가 분명한가?
- 배터리, 알림 피로, 오작동이 있는가?

### P3 — Production 롤아웃

| 단계 | 대상 | 기준 |
| --- | --- | --- |
| 1차 | 한국 10% | crash/ANR, 권한 거부율, 리뷰 확인 |
| 2차 | 한국 50% | 치명적 알림 실패·정책 경고 없음 |
| 3차 | 한국 100% | 7일 안정화 |
| 4차 | 영문 리스팅 추가 후 국가 확장 | 영문 도움말·개인정보처리방침 준비 |

데이터를 수집하지 않는 v1에서는 통상적인 퍼널 분석 대신 Play Console의 crash/ANR, 설치·삭제 추세, 평점·리뷰, 비공개 테스터 설문으로 품질을 본다.

## 7. 심사 제출 체크리스트

### Play Console

- [ ] 개발자 신원·연락처·결제 프로필 확인
- [ ] 앱 생성: 앱, 무료, 건강 및 피트니스
- [ ] 기본 스토어 리스팅: 이름, 짧은/긴 설명, 이메일, 개인정보처리방침 URL
- [ ] 앱 아이콘, feature graphic, 휴대전화 스크린샷
- [ ] App access: 제한 없음
- [ ] Ads: 없음
- [ ] Content rating: IARC 설문 제출
- [ ] Target audience: 성인 직장인 중심
- [ ] Data safety: 수집·공유 없음 (최종 SDK 검사 후 제출)
- [ ] Health apps declaration: 웰니스 기능에 맞춰 정확히 제출
- [ ] `USE_FULL_SCREEN_INTENT` 유지 시 해당 선언 및 심사용 동작 설명/영상 제출
- [ ] 국가/지역: 대한민국
- [ ] 가격: 무료

### 빌드와 QA

- [ ] upload key로 서명된 AAB
- [ ] release 빌드에서 개발 메뉴·개발 테스트 패널 비노출
- [ ] 디버그 keystore 미사용
- [ ] `versionCode` 증가
- [ ] Play App Signing 활성화
- [ ] 알림/정확한 알람/전체 화면 권한의 거부·재허용 경로 검증
- [ ] 앱이 완전히 종료되어도 정규 알림이 오는지 검증
- [ ] 기기 재부팅 후 예약 복구 여부 검증
- [ ] Android 12~16 실기기 또는 pre-launch report 확인
- [ ] 아이콘·feature graphic의 정확한 픽셀/포맷/alpha 검수

## 8. 권장 릴리스 순서

```text
정책 결정(전체 화면) → 권한 최소화 → release signing/AAB
→ 내부 테스트 → 스토어 리스팅·정책 양식 완료
→ 비공개 테스트(필요 시 12명·14일) → production 접근 신청
→ 한국 10% 단계 롤아웃 → 100%
```
